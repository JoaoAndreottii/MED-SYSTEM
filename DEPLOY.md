# Med-System Deployment Guide

## Railway Deployment

This guide covers deploying Med-System to **Railway** (modern PaaS platform).

### Prerequisites

1. **Railway Account** — Sign up at https://railway.app
2. **Railway CLI** — Install from https://docs.railway.app/cli/install
3. **Git Repository** — Initialized and pushed to GitHub/GitLab
4. **Environment Variables** — Prepared for production

### Step 1: Connect Repository to Railway

```bash
# Login to Railway
railway login

# Link your project
cd med-system
railway init

# Select your Git repository
# Choose: Connect to GitHub
# Select: med-system repo
```

### Step 2: Configure Environment Variables

In Railway Dashboard:
1. Go to **Project** → **Variables**
2. Add the following:

```
# Required
NODE_ENV=production
JWT_SECRET=<generate-strong-32-char-secret>
DATABASE_URL=<auto-generated-by-railway>
REDIS_HOST=<auto-generated-by-railway>
REDIS_PORT=6379

# Optional (if using external APIs)
OPENAI_API_KEY=<your-key>
TWILIO_ACCOUNT_SID=<your-sid>
TWILIO_AUTH_TOKEN=<your-token>
RESEND_API_KEY=<your-key>

# Application
LOG_LEVEL=info
QUEUE_CONCURRENCY=5
```

### Step 3: Deploy

```bash
# Deploy using Railway CLI
railway up

# Or push to GitHub and auto-deploy (if GitHub integration enabled)
git push origin main
```

### Step 4: Run Migrations

After first deploy:

```bash
# Option A: Via Railway CLI
railway run npm run prisma:migrate -- --skip-generate

# Option B: Via Railway Dashboard
# Go to service → "Details" → "Run command"
# Run: npm run prisma:deploy
```

### Step 5: Verify Deployment

```bash
# Check service status
railway status

# View logs
railway logs

# Test API
curl https://your-railway-app.up.railway.app/health
```

## Scaling Guidelines

### Vertical Scaling (More Power)
1. Go to **Service** → **Settings** → **Resources**
2. Increase **Memory** or **CPU**
3. Service restarts automatically

```
Development:  256MB Memory, 0.5 vCPU
Staging:      512MB Memory, 1.0 vCPU
Production:   1GB Memory, 1.0 vCPU (or more)
```

### Horizontal Scaling (Multiple Instances)
1. Go to **Service** → **Settings** → **Deployment**
2. Increase **Replicas** (2-3 recommended)
3. Railway auto-load-balances

```
# Load balancer will distribute requests
Replica 1: api-1.railway.app
Replica 2: api-2.railway.app
Replicas load-balanced automatically
```

### Database Scaling
- PostgreSQL: Increase storage in **Railway Dashboard** → **Database** → **Settings**
- Redis: Increase memory for more queue capacity

## Continuous Deployment

### Automatic Deployment on Push

Enable in Railway Dashboard:
1. **Project** → **Settings** → **Deploy on push**
2. Select branch (e.g., `main`)
3. Every push triggers automatic rebuild & deploy

### Manual Deployment

```bash
# Trigger deploy via CLI
railway trigger

# Or push to GitHub (if auto-deploy enabled)
git push origin main
```

### Rollback

```bash
# View deployment history

# Rollback to previous version
railway rollback <deployment-id>
```

## Common Issues & Solutions

### Issue: Deploy Fails with "npm install" Error
**Solution:**
```bash
# Ensure package-lock.json is committed
git add package.json package-lock.json
git commit -m "lock: update npm dependencies"
git push origin main
```

### Issue: Database Connection Refused
**Solution:**
1. Check `DATABASE_URL` in Railway Dashboard
2. Ensure PostgreSQL service is running
3. Run migrations: `railway run npm run prisma:migrate`
4. Check logs: `railway logs -s database`

### Issue: Redis Connection Failed
**Solution:**
1. Verify `REDIS_HOST` and `REDIS_PORT` in Railway
2. Ensure Redis service is healthy
3. Check Redis memory usage (may be full)
4. Restart Redis: **Service** → **Settings** → **Restart**

### Issue: Out of Memory
**Solution:**
1. Increase memory limit in **Service** → **Settings**
2. Reduce `QUEUE_CONCURRENCY` to lower worker load
3. Check for memory leaks in application logs
4. Consider adding another replica for load distribution

### Issue: Slow Database Queries
**Solution:**
1. Add indexes to frequently queried columns (already in schema)
2. Check query patterns with `npm run prisma:studio`
3. Enable query logging: `LOG_LEVEL=debug`
4. Consider database scaling (increase resources)

## Monitoring & Logging

### View Logs
```bash
# Real-time logs
railway logs -f

# Filter by service
railway logs -s api
railway logs -s database
railway logs -s redis

# Search for errors
railway logs | grep -i error
```

### Health Check Endpoint

Implement a health check in your API:

```typescript
// src/routes/health.ts
router.get('/health', (req, res) => {
  res.json({
    status: 'ok',
    timestamp: new Date(),
    uptime: process.uptime(),
  })
})
```

Railway will periodically call this to ensure service is alive.

### Metrics to Monitor

- **Response Time** — Average, P95, P99
- **Error Rate** — 5xx errors per minute
- **Database Connections** — Should stay constant
- **Memory Usage** — Should not exceed limit
- **CPU Usage** — Should peak under load

## Database Backups

Railway provides automatic backups:
- **Retention:** 7 days (default)
- **Frequency:** Daily
- **Recovery:** Via Railway Dashboard → Database → Backups

To export data:
```bash
# Backup to local file
pg_dump $DATABASE_URL > backup.sql

# Restore from backup
psql $DATABASE_URL < backup.sql
```

## Security Considerations

### API Security
- [x] HTTPS enforced (Railway default)
- [x] CORS configured
- [x] Rate limiting (recommended)
- [x] Input validation (Zod)
- [x] JWT tokens (secure)

### Database Security
- [x] PostgreSQL password protected
- [x] Private network (Redis/DB not exposed)
- [x] Credentials in environment variables
- [ ] SSL/TLS for database connections (enable)

### Secrets Management
- Never commit `.env` to Git
- Use Railway Dashboard for secrets
- Rotate JWT_SECRET regularly
- Use strong passwords (32+ chars)
